# Shift Assignment Preview Parity Guide

## Phase 10 Boundary

Compare the existing `public/shifting-assignment.php` page with the unlinked
`public/shift-assignment-react-preview.php` pilot before any replacement.

This phase does not approve navigation changes, writes, new mutation APIs,
schema changes, data resets, or removal of the legacy page. `public/calendar.php`
remains the zero-mistake visual reference and is not changed.

## Required Test Setup

Use a disposable or approved non-production environment with the role fixtures
listed in `docs/shift-assignment-role-test-matrix.md`. Build the preview first:

```bash
cd frontend
npm run build:preview
cd ..
```

Choose a range containing known assignments, preferably including a Shift 3,
overtime or holiday assignment, and an existing warning. Record the range in
ISO format and use exactly the same boundaries on both pages.

Do not create, edit, copy, resize, confirm, dismiss, export, or delete data
during parity testing.

## Automated Non-Mutating Gate

Run:

```bash
php tests/shift-assignment-preview-parity.php
```

The gate checks:

- both PHP pages require the normal authentication shell and `shifts.view`;
- the preview retains its unique route identity and dedicated React root;
- production navigation still links only to `shifting-assignment.php`;
- the React data client uses only the three approved GET resources;
- both v1 Shift Assignment routes require GET and `shifts.view`;
- default role grants and service scope rules remain characterized;
- no preview or React API client write method is introduced.

This source-level gate does not prove browser layout or live database parity.

## Role-Based Browser Procedure

Repeat for Super Admin, Admin, Supervisor, Agent, and Intern. Repeat Supervisor
with no division when a safe fixture is available.

- [ ] Log in normally and complete 2FA when required.
- [ ] Record effective Shift Assignment permissions and expected scope.
- [ ] Open `/shifting-assignment.php`.
- [ ] Open `/shift-assignment-react-preview.php` in a second tab.
- [ ] Confirm neither page exposes a raw error, stack trace, server path, or SQL.
- [ ] Select the same date range and view.
- [ ] Compare daily assignment IDs, agents, dates, shift ranges, types, statuses,
      overtime flags, and holiday flags.
- [ ] Compare weekly assignment IDs and workload totals.
- [ ] Compare monthly assignment IDs when the selected month has data.
- [ ] Compare agent and division filter IDs against the expected role scope.
- [ ] Compare assignment type and status options.
- [ ] Confirm a filter cannot expand data beyond the server-provided scope.
- [ ] Compare jumpshift, conflict, and coverage warning types and counts.
- [ ] Confirm holidays and overtime indicators refer to the same records.
- [ ] Confirm empty results do not imply data deletion.
- [ ] Confirm the preview contains no working add, edit, delete, copy, template,
      holiday, overtime, confirm, dismiss, or export action.
- [ ] Confirm the browser console has no errors.
- [ ] Confirm all API calls are the approved protected GET routes.

## Data Parity Rules

Pass only when the same scoped source records produce compatible read results:

- [ ] Shift 1 is `00:00-08:00` and 480 minutes.
- [ ] Shift 2 is `08:00-16:00` and 480 minutes.
- [ ] Shift 3 displays `16:00-24:00`, stores next-day `00:00`, and remains
      cross-day.
- [ ] API dates remain ISO `YYYY-MM-DD`.
- [ ] Display dates remain `dd-mm-yyyy`.
- [ ] Daily, weekly, and monthly ranges include overlapping cross-day records
      consistently.
- [ ] Assignment counts and IDs match for the same role, scope, range, and
      supported filters.
- [ ] Scheduled agent counts and total minutes can be traced to returned rows.
- [ ] Workload recap does not contradict the legacy service calculation.
- [ ] Overtime and holiday counts match the returned assignment flags.
- [ ] Warnings are generated by existing PHP service rules and expose no
      sensitive fields.
- [ ] Agent payloads omit email, credentials, 2FA fields, notes, and internal
      actor IDs.

Legacy-only free-text, holiday-only, risk-chip, audit, and configuration
controls are not React parity requirements for this read-only pilot. Record
them as deferred gaps rather than defects unless they change the underlying
scoped records being compared.

## Known Preview Differences To Record

- The v1 assignments API supports an allowlisted role query after applying
  server scope, but the React candidate does not expose it because the context
  reports `role_filter_supported: false` and the legacy page has no role filter.
- The preview summary cards are a smaller read-only subset of the legacy
  insights. Compare source counts and minutes, not card labels or card count.
- Audit, configuration, write actions, free-text search, holiday-only filters,
  and CSV export remain legacy-only by design.

## Visual And Responsive Review

Use `calendar.php` as the visual source of truth.

- [ ] Page spacing, section gaps, and card padding follow the Calendar rhythm.
- [ ] Toolbar and filter density remain operational rather than oversized.
- [ ] Summary cards remain readable without dominating the page.
- [ ] Desktop table headers and rows are legible and not clipped.
- [ ] Tablet layout preserves filters and warnings without overlap.
- [ ] Mobile hides the wide table and renders readable assignment cards.
- [ ] Loading state identifies what is loading.
- [ ] Empty state clearly says no scoped records matched.
- [ ] `401`, `403`, `422`, and network errors use safe actionable messages.
- [ ] Dark mode has sufficient text, border, status, and focus contrast.
- [ ] Keyboard focus is visible on tabs, filters, and buttons.
- [ ] Zoom at 200% does not hide content or create unreachable controls.

Recommended viewport evidence:

- desktop: 1440 x 900;
- laptop: 1280 x 800;
- tablet: 768 x 1024;
- mobile: 390 x 844.

## Security Checks

- [ ] Unauthenticated preview page redirects to `/login.php`.
- [ ] Unauthenticated v1 context and assignments calls return `401`.
- [ ] Authenticated users without `shifts.view` receive generic page denial and
      API `403`.
- [ ] Expired session returns to authentication and APIs return `401`.
- [ ] POST, PATCH, and DELETE against v1 read resources return `405`.
- [ ] Invalid ranges and filters return `422` with the five-key envelope.
- [ ] Responses contain `meta.request_id`.
- [ ] No response contains raw exception text, SQL, filesystem paths,
      credentials, environment values, email, password, or 2FA secrets.
- [ ] `X-Powered-By` remains absent where the established security rule applies.
- [ ] GET parity checks do not change assignment or audit table counts.

## Result Classification

- **Pass:** scoped record identity, totals, security, and required visual states
  are compatible.
- **Fail:** the preview omits, adds, or changes an in-scope record; weakens
  permission scope; leaks data; activates a write; or has a blocking UI defect.
- **Deferred gap:** a legacy-only feature intentionally outside this read-only
  phase.
- **Blocked:** required fixture, known data range, or authenticated browser
  environment is unavailable. Record the missing prerequisite.

Production replacement is not approved until every required role has recorded
evidence and all security/data failures are resolved.

## Phase 11 Limited Internal Pilot

Pilot objective: collect controlled read-only feedback without replacing the
legacy page or exposing the preview broadly.

- Allowed role: exact `super_admin` only.
- Required permission: `shifts.view`.
- Access: direct URL `/shift-assignment-react-preview.php`; no navigation link.
- Denial: non-Super Admin users receive the existing safe generic page denial.
- APIs: remain independently authenticated, `shifts.view` protected, and
  server-scoped. The page gate does not weaken API rules.
- Limitations: no writes, audit view, configuration, export, free-text search,
  production navigation, or replacement approval.
